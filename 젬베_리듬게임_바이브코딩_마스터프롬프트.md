# 🥁 젬베 리듬 게임 - 바이브코딩 마스터 프롬프트
> **목표:** 단일 HTML 프로토타입 → 상업 수준의 모바일 친화적 PWA 리듬 게임으로 업그레이드  
> **권장 AI:** Claude Opus 4.7 / Claude Code / Cursor  
> **권장 흐름:** 각 Phase의 프롬프트를 **순서대로** 복붙 → 결과물 확인 → 다음 Phase로 진행

---

## 📋 사용 방법

1. **Phase 0**은 모든 대화 세션 시작 시 1번 입력 (컨텍스트 셋업)
2. **Phase 1 → Phase 10** 순서대로 진행 (건너뛰지 말 것)
3. 각 Phase 끝의 **"검증 체크리스트"**가 통과되어야 다음 Phase 진행
4. 막힌 부분은 **"🔧 디버깅 프롬프트"** 섹션 참조
5. AI가 멋대로 다른 부분을 수정하면 **"방금 수정한 X 외에 다른 코드는 절대 건들지 마"** 강조

---

# 🎯 Phase 0 — 프로젝트 컨텍스트 셋업

**언제 사용:** 새 채팅을 시작할 때 가장 먼저 입력. 이후 모든 Phase 프롬프트보다 앞에 위치.

```
너는 HTML5 Canvas, Web Audio API, 모던 JavaScript(ES2022+)에 능숙한 리듬 게임 전문 개발자야.
지금부터 함께 "젬베(Djembe) 리듬 게임"을 단계적으로 업그레이드할 거야.

[게임 컨셉]
- 서아프리카 전통 타악기 "젬베"를 소재로 한 모바일 친화적 웹 리듬 게임
- 3개 타격 영역: SLAP(왼쪽, 'D'/왼쪽터치) · BASS(중앙, 'Space'/가운데터치) · TONE(오른쪽, 'K'/오른쪽터치)
- 위에서 떨어지는 노트가 판정선에 닿는 순간 키/터치 입력 → 사운드 재생 + 판정
- BGM 끝나면 결과창 → 점수, 콤보, 정확도, 등급(S/A/B/C/D) 표시

[기술 스택]
- Vanilla JS (프레임워크 없음, 의존성 최소화)
- Web Audio API (오디오 정밀 타이밍)
- Canvas 2D (게임 렌더링)
- LocalStorage (점수/설정 저장)
- PWA 지원 (오프라인 실행 가능)
- 반응형 (데스크탑 1920x1080 → 모바일 360x640 모두 대응)

[코드 품질 원칙]
1. 단일 파일이 아닌 **모듈 분리** (game/, audio/, ui/, assets/ 디렉토리)
2. 클래스 기반 OOP + 명확한 책임 분리
3. **모든 타이밍은 audioContext.currentTime 기준** (절대 Date.now / requestAnimationFrame 시간 신뢰 X)
4. 60fps 유지 (Performance.now() 측정 후 프레임 드랍 시 경고)
5. 접근성: 키 리매핑, 색약 모드, 자막 가능 구조
6. 한국어 우선, 영어 토글 가능 (i18n 구조)

[작업 규칙]
- 한 번에 한 Phase씩만 진행
- 코드 변경 시 **변경된 파일 전체**를 보여줄 것 (diff 아니라 전체)
- 새 파일 생성 시 디렉토리 트리도 함께 보여줄 것
- 의사결정이 필요한 부분은 옵션 2~3개를 제시하고 사용자에게 물어볼 것
- 끝에 항상 "다음 Phase로 진행할까요?" 묻고 멈출 것

준비됐으면 "준비 완료. Phase 1 프롬프트를 입력해주세요"라고만 답해.
```

**✅ 검증:** AI가 위 메시지에 "준비 완료" 응답을 했다면 OK. 다른 코드를 먼저 짜려고 하면 다시 강조.

# ⏱️ Phase 2 — 정밀 타이밍 엔진 (Conductor)

**언제 사용:** Phase 1 통과 후. 리듬 게임의 심장인 오디오 동기화 시스템 구현.

## 🎯 목표
- "노트가 떨어지는 시각"과 "BGM 박자"가 절대 어긋나지 않게 함
- 디바이스별 오디오 지연 보정(Calibration) 시스템

## 📦 입력 프롬프트

```
[Phase 2 - 정밀 타이밍 엔진]

리듬 게임에서 가장 중요한 건 "타이밍 정확도"야. 일반적인 setInterval이나 requestAnimationFrame의 
delta time을 사용하면 박자가 미세하게 어긋나서 사용자가 곡과 안 맞다고 느껴.

audio/Conductor.js를 다음 사양으로 구현해줘:

[Conductor 클래스 사양]
- 생성자: AudioContext, bpm, offset(곡 시작 전 무음 ms) 받음
- start(audioBuffer): BGM 재생 시작, audioContext.currentTime 기준 startTime 저장
- get songPositionInSeconds(): 현재 곡의 재생 시간(초) 반환 = audioContext.currentTime - startTime
- get songPositionInBeats(): 현재 박자 위치 = songPositionInSeconds * (bpm / 60)
- get currentBeat(): Math.floor(songPositionInBeats)
- onBeat(callback): 박자가 바뀔 때마다 콜백 실행 (메트로놈, 비주얼 펄스용)
- pause() / resume()
- 일시정지 시 정확한 시간 보존 (다시 재생할 때 어긋남 없게)

[Calibration 시스템 - utils/CalibrationManager.js 추가 생성]
디바이스마다 오디오 출력 지연이 다르기 때문에(블루투스는 200ms 이상), 
사용자가 설정 화면에서 보정할 수 있어야 함:

1. AudioOffset (오디오 오프셋): 사운드 재생 명령 → 실제 들리기까지의 지연
2. VisualOffset (비주얼 오프셋): 화면 갱신 지연 (대부분 0)
3. InputOffset (입력 오프셋): 사용자가 박자보다 일찍/늦게 누르는 경향 보정

캘리브레이션 화면 흐름:
- 메트로놈이 4박자씩 반복 재생
- 사용자가 박자에 맞춰 스페이스바 입력
- 16번 입력 평균 = inputOffset
- LocalStorage에 저장

[Phase 2에서 구현할 것]
1. audio/AudioEngine.js: AudioContext 초기화, 사용자 첫 인터랙션 시 resume()
2. audio/SampleBank.js: fetch → decodeAudioData → 캐싱, playSample(name, time, volume) 메소드
3. audio/Conductor.js: 위 사양대로
4. utils/CalibrationManager.js: 위 사양대로
5. scenes/CalibrationScene.js: 캘리브레이션 UI (메트로놈 + 진행률 + 결과 표시)

[테스트 방법]
- TitleScene에 "캘리브레이션" 임시 버튼 추가
- 캘리브레이션 완료 시 콘솔에 "Input offset: XXms" 출력
- BGM 없이 SampleBank로 60BPM 4박자 메트로놈만 재생되도 박자가 안 어긋나는지 확인

코드 짜기 전에, "왜 Date.now() 대신 audioContext.currentTime을 써야 하는지"를 
3줄로 먼저 설명해줘. 그 다음 코드.
```

## 🔧 디버깅 프롬프트 (만약 박자가 어긋난다면)

```
재생 5분쯤 지나면 박자가 미세하게 밀려. 다음을 점검해줘:
1. Conductor의 startTime이 audioContext.currentTime인지
2. requestAnimationFrame delta가 아닌 songPositionInSeconds로 노트 위치 계산하는지
3. BGM과 sample 재생 시 audioContext.currentTime + offset 방식으로 스케줄링하는지
원인 찾으면 수정 후 변경된 파일만 보여줘.
```

## ✅ 검증 체크리스트
- [ ] 60BPM 메트로놈을 1분간 들어도 박자 어긋남 없음
- [ ] 일시정지 후 재생 시 박자 끊김 없음
- [ ] 캘리브레이션 후 inputOffset이 LocalStorage에 저장됨
- [ ] 모바일에서도 정상 동작 (iOS Safari는 사용자 터치 후에만 오디오 활성)

# 🎼 Phase 3 — 노트 차트(Beatmap) 시스템

**언제 사용:** Phase 2 통과 후. 곡에 맞춰 노트가 떨어지는 차트 데이터 구조 설계.

## 🎯 목표
- JSON 기반 차트 포맷 정의 → 다양한 곡 추가 용이
- 노트 종류 확장 (탭/롱노트/롤/동시타)

## 📦 입력 프롬프트

```
[Phase 3 - 노트 차트 시스템]

리듬 게임의 콘텐츠는 "채보(Chart)"야. 각 곡마다 어느 시점에 어떤 노트가 떨어질지 정의해야 해.

[차트 JSON 포맷 - assets/charts/{songId}.json]
{
  "songId": "tutorial_01",
  "title": "처음 만나는 젬베",
  "artist": "Djembe Master",
  "bpm": 90,
  "offset": 1000,
  "duration": 60000,
  "difficulty": {
    "easy": 1,
    "normal": 3,
    "hard": 5
  },
  "audioFile": "assets/sounds/bgm/tutorial_01.mp3",
  "previewStart": 8000,
  "background": "savanna",
  "notes": {
    "easy": [
      { "time": 2000, "lane": "bass", "type": "tap" },
      { "time": 3000, "lane": "bass", "type": "tap" },
      { "time": 4000, "lane": "slap", "type": "tap" },
      { "time": 5000, "lane": "tone", "type": "hold", "duration": 1000 }
    ],
    "normal": [ ... ],
    "hard": [ ... ]
  }
}

[노트 타입]
- "tap": 한 번 누르고 떼기 (기본)
- "hold": 누르고 일정 시간 유지 (롱노트) - 시작/끝 모두 판정
- "roll": 짧은 시간 안에 여러 번 연타 (Bass 트레몰로 같은 표현)
- "chord": 같은 time에 다른 lane의 노트가 동시에 = 동시타 (UI 자동 처리)

[구현할 것]
1. game/ChartLoader.js
   - loadChart(songId): fetch → JSON parse → 검증
   - getNotes(difficulty): 해당 난이도의 노트 배열 반환 (time 오름차순 정렬)

2. game/Note.js
   - 속성: time, lane('slap'|'bass'|'tone'), type, duration, state('upcoming'|'active'|'hit'|'missed')
   - getYPosition(currentTime, scrollSpeed, judgmentY): 현재 화면 Y 좌표 계산
   - 화면 밖이면 cull, 판정선 ±100ms는 'active' 상태로 전환
```

# ⚖️ Phase 4 — 판정 & 점수 시스템

**언제 사용:** Phase 3 통과 후. 입력과 노트를 매칭하여 판정.

## 🎯 목표
- 4단계 판정 (Perfect/Great/Good/Miss) + 정확한 판정 윈도우
- 콤보, 스코어, 정확도(Accuracy%), 등급(S~D) 계산

## 📦 입력 프롬프트

```
[Phase 4 - 판정 & 점수 시스템]

지금까지 노트가 떨어지기만 했는데, 이제 입력 → 판정 → 점수 시스템을 만든다.

[판정 윈도우 (ms 단위, 노트 시간 기준 ±)]
- Perfect: ±30ms  (점수 1000, 정확도 100%)
- Great:   ±60ms  (점수 700,  정확도 75%)
- Good:    ±100ms (점수 300,  정확도 50%)
- Miss:    >100ms 또는 노트가 판정선 + 100ms 지났는데 입력 없음 (점수 0, 정확도 0%, 콤보 끊김)

[InputManager 강화]
- 키 다운 이벤트 발생 시점의 audioContext.currentTime을 정확히 캡처
- 캘리브레이션의 inputOffset 적용: 보정된 시각 = currentTime - inputOffset
- onInput(lane, timestamp) 콜백을 PlayScene에 전달

[JudgmentSystem 구현]
- handleInput(lane, inputTime, activeNotes):
  1. 같은 lane의 활성 노트 중 |note.time - inputTime|이 가장 작은 것 찾기
  2. 시간 차이로 Perfect/Great/Good/Miss 결정
  3. 결과 객체 반환: { judgment, deltaMs, noteId }
- checkMissedNotes(currentTime, activeNotes): 판정선 + 100ms 지난 미입력 노트 → Miss 처리

[ScoreSystem 구현]
- 누적 점수 = Σ(노트별 점수 × 콤보 보너스)
- 콤보 보너스: 50콤보마다 +10% (최대 +100%)
- 정확도 = (Σ 노트별 정확도) / 총 노트 수 × 100%
- 등급:
  S: 95%+
  A: 90~94%
  B: 80~89%
  C: 70~79%
  D: 70% 미만
- "올퍼펙트(Full Combo / All Perfect)" 플래그도 별도 관리

[HUD 업데이트]
- 좌측 상단: SCORE (12자리, 0 패딩, 우측 정렬)
- 우측 상단: COMBO (숫자가 클수록 폰트 커지고 펄스 애니메이션)
- 중앙 상단: 정확도 % (실시간)
- 노트 히트 시 판정선 위에 큰 글자로 "PERFECT!" "GREAT!" "GOOD!" "MISS" 0.3초 표시
- 색상은 Theme.js의 PERFECT/GREAT/GOOD/MISS 사용

[실패 처리]
- 미스 5연속: 화면 흔들기 (CSS transform shake)
- 미스 10연속: BGM 일시 디튠(detune) -100 cents → 200ms 후 복원

[디버그 표시]
- 우측 하단에 작게 (개발 모드만):
  "Last hit: PERFECT (+12ms)" 같은 식으로
- localStorage.debug = true 일 때만 표시

[중요한 디테일]
- 같은 시간에 lane이 다른 노트 2개(chord)일 때, 각각 독립 판정
- hold 노트는 시작 판정 + 끝 판정 따로 (시작 Perfect, 끝 Good 같은 식)
- roll 노트는 시작 시점부터 duration 동안 연타 횟수에 따라 점수

전체 구현 후 다음 사항이 동작하는지 자체 점검 후 보여줘:
1. 정확히 박자에 누르면 콘솔에 "PERFECT 0ms" 찍히는지
2. 일부러 50ms 늦게 누르면 "GREAT +50ms"
3. 안 누르고 지나가면 "MISS"
4. 100콤보 달성 시 화면에 "100 COMBO!" 1초간 큰 글씨
```

# 🔧 트러블슈팅 프롬프트 모음

각 상황별로 즉시 사용 가능한 디버깅 프롬프트.

## A. 박자가 안 맞아요

```
플레이 중 BGM과 노트 떨어지는 박자가 점점 어긋나. 다음 순서로 점검 후 수정해줘:
1. Conductor의 startTime이 정확히 audioContext.currentTime인지
2. PlayScene의 update()에서 currentTime을 audioContext.currentTime에서 받는지 (performance.now() 사용 시 BUG)
3. Note의 Y 좌표 계산이 (note.time - currentTime) * scrollSpeed인지
4. BGM 재생 시 audioSource.start(0) 했는지 (start() 만 하면 0 기본값이지만 명시 권장)
원인 찾으면 변경된 파일만 보여줘.
```

## B. 모바일에서 사운드가 안 나요

```
모바일 Safari에서 게임 시작해도 소리가 안 나. 다음 확인:
1. AudioContext는 사용자 제스처 후에만 활성화 가능
2. 시작 버튼 클릭 핸들러에서 audioContext.resume() 호출하는지
3. 자동재생 정책 우회: 첫 클릭에서 1샘플의 무음 재생 트릭 사용했는지
원인 수정 후 InputManager + AudioEngine 보여줘.
```

## C. 노트 떨어지는 게 끊겨요 (프레임 드랍)

```
복잡한 차트(노트 50개 동시)에서 프레임 드랍 발생. 다음 최적화:
1. 화면 밖 노트는 그리지 않기 (culling)
2. 파티클 최대 개수 제한 (500개)
3. 객체 풀링 적용 (Note, Particle 모두)
4. 배경 등 정적 요소는 offscreen canvas로 1회만 그리기
5. requestAnimationFrame 콜백 안에서 매번 새 객체 생성하지 않기
변경된 파일만 보여주고, 어떤 부분이 병목이었는지 1줄 설명 추가.
```

## D. 판정이 너무 짜요/후해요

```
지금 판정 윈도우가 너무 [짭니다/후합니다]. 다음과 같이 조정해줘:
- Perfect: [현재 30ms → 새 값]ms
- Great: [현재 60ms → 새 값]ms  
- Good: [현재 100ms → 새 값]ms
JudgmentSystem.js의 WINDOW 상수만 수정. 다른 파일 건들지 마.
```

## E. 특정 노트가 안 보여요

```
[BASS/SLAP/TONE] 노트만 화면에 안 나타나. 다음 점검:
1. assets/images/note-[이름].png 파일 존재 확인
2. AssetLoader에 등록되었는지
3. Note.js의 getSprite()가 lane으로 올바른 이미지 반환하는지
4. Canvas drawImage 호출 시 이미지가 로드 완료되었는지 (image.complete)
원인 찾고 변경 파일만 보여줘.
```
